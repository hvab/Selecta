# Спека проекта: генератор тем для Эгеи

## 1. Цель проекта

Нужно сделать браузерный конструктор тем для блогового движка Эгея.

Пользователь открывает страницу, настраивает внешний вид темы в визуальном интерфейсе, сразу видит результат на полноценном превью блога, а затем скачивает готовую папку темы, которую можно загрузить на хостинг и выбрать в настройках Эгеи.

Главная идея проекта:
**дать человеку возможность сделать аккуратную собственную тему для Эгеи без ручного редактирования CSS и PHP-файлов.**

В первой версии важнее не максимальная свобода кастомизации, а надёжный путь:
**открыл → настроил → скачал → установил → получил ровно то, что видел в превью.**

## 2. Общая концепция

Эгея уже поддерживает наследование тем.

Базовая тема `plain` содержит основную разметку, стили, типографику и поведение. Остальные обычные темы в основном наследуются от неё и переопределяют CSS-переменные, а также метаданные в `theme-info.php`.

Поэтому первая версия конструктора должна генерировать не полностью самостоятельную тему, а **дочернюю тему на основе `plain`**.

Генератор должен работать с единой моделью настроек темы. Из одного и того же состояния должны строиться:

- визуальное превью в браузере;
- файл `styles/main.css`;
- файл `theme-info.php`;
- итоговый архив для скачивания.

Важно, чтобы превью и скачиваемая тема не расходились.

## 3. Основной пользовательский сценарий

1. Пользователь открывает генератор.
2. Видит превью блога в стиле Эгеи.
3. Меняет цвета, шрифты и несколько типографических параметров.
4. Сразу видит изменения в превью.
5. Вводит название темы.
6. Нажимает кнопку скачивания.
7. Получает архив с папкой темы.
8. Загружает папку в каталог пользовательских тем Эгеи.
9. Выбирает тему в настройках движка.
10. Видит на своём блоге тот же внешний вид, который был в генераторе.

Этот путь должен быть коротким и очевидным. Пользователь не должен понимать внутреннее устройство Эгеи, чтобы получить рабочую тему.

## 4. Что именно генерирует проект

В MVP генератор должен создавать архив с такой структурой:

```text
theme-name/
  theme-info.php
  styles/
    main.css
```

Где:

- `theme-info.php` описывает тему для Эгеи;
- `styles/main.css` содержит переопределения CSS-переменных поверх `plain`.

Тема должна:

- иметь `based_on` = `plain`;
- иметь корректное имя для отображения;
- содержать цвета для списка тем в настройках Эгеи;
- работать без собственных шаблонов, JS, изображений и шрифтов;
- использовать всё необходимое от родительской темы `plain`.

## 5. Базовые допущения первой версии

В первой версии считаем, что:

1. Генерируются только темы, наследуемые от `plain`.
2. Пользователь получает ZIP-архив с папкой темы внутри, а не отдельные файлы по одному.
3. Генератор работает как отдельный статический веб-проект без связи с конкретной установкой Эгеи.

Эти ограничения сознательные. Они позволяют сначала сделать полезный и надёжный инструмент, а не сразу полноценную среду разработки тем.

## 6. Что должно быть в MVP

### 6.1. Живое превью

На странице должно быть полноценное превью блога, достаточное для проверки темы.

Минимально в превью нужны:

- заголовок блога;
- подзаголовок;
- основное меню;
- заголовок заметки;
- основной текст заметки;
- лид;
- обычные ссылки;
- посещённые ссылки;
- тег или вторичная ссылка;
- заметный hover-state;
- цитата или выделенный фрагмент;
- форма или хотя бы поля ввода;
- footer.

Превью должно показывать не абстрактные цветовые квадраты, а именно типичный блог Эгеи.

### 6.2. Настройка темы

В MVP пользователь должен иметь возможность настроить:

#### Основные цвета

- фон;
- основной текст;
- заголовки;
- ссылки;
- посещённые ссылки;
- hover;
- вторичный текст.

#### Типографику

- шрифт интерфейса;
- шрифт основного текста заметок;
- размер основного текста;
- масштаб заголовков;
- межстрочный интервал основного текста.

#### Несколько общих параметров композиции

- максимальную ширину контента;
- ширину боковых отступов;

В MVP не нужно выводить все переменные `plain`. Нужно дать только те настройки, которые человек действительно понимает и которые заметно меняют характер темы.

### 6.3. Название темы

Пользователь должен ввести:

- техническое имя папки темы;
- отображаемое имя темы.

Желательно, чтобы техническое имя автоматически предлагалось из отображаемого, но пользователь мог его поправить.

### 6.4. Скачать тему

Пользователь должен получить готовый архив по одной кнопке.

Архив должен содержать папку темы, а не просто россыпь файлов в корне архива.

Имя архива должно быть понятным, например:

```text
my-theme.zip
```

### 6.5. Проверка перед скачиванием

Перед генерацией архива нужно проверить как минимум:

- заполнено ли отображаемое имя;
- задано ли допустимое имя папки;
- нет ли недопустимых символов в имени папки;
- есть ли минимально необходимые значения;
- согласованы ли цвета в `theme-info.php` с тем, что реально используется в CSS.

Если проверка не проходит, пользователь должен получить понятное объяснение, что нужно исправить.

## 7. Модель темы

В проекте должна быть явная модель состояния темы.

Примерно по смыслу она должна содержать:

```text
meta
  folderName
  displayName
  basedOn
  supportsDarkMode
  useLikelyLight

palette
  background
  foreground
  headings
  link
  linkVisited
  hover
  tag
  engineText
  admin
  active
  markedTextBackground
  inputBackground
  inputText

typography
  mainFontFamily
  noteFontFamily
  noteTextSize
  noteTextLineHeight
  titleScale

layout
  maxWidth
  margins
```

Не обязательно именно с такими именами в коде, но смысловые группы должны существовать.

Из этой модели должны строиться:

- CSS-переменные;
- значения `colors` в `theme-info.php`;
- превью;
- сериализуемое состояние для будущего сохранения и шаринга.

## 8. Принцип генерации CSS

Генерируемый `styles/main.css` не должен дублировать всю тему `plain`.

Он должен содержать только осмысленные переопределения поверх родителя.

Примерная структура:

```css
:root {
  --backgroundColor: ...;
  --foregroundColor: ...;
  --headingsColor: ...;
  --linkColor: ...;
  ...
}
```

Если какие-то значения являются производными от основных цветов, генератор должен сам вычислять их и записывать в CSS.

Например:

- прозрачный фон листов;
- underline-цвета;
- thin rule;
- secondary text;
- возможно, часть hover/visited оттенков.

Пользователь должен задавать понятные исходные значения, а не ручками собирать десятки технических переменных.

## 9. Принцип генерации `theme-info.php`

Файл должен содержать как минимум:

- `display_name`;
- `colors`;
- `based_on` = `plain`;
- `meta_viewport`;
- `supports_dark_mode`.

В MVP достаточно одного display name без локализаций.

Сам интерфейс генератора в MVP ведём только на английском языке.
Мультиязычность интерфейса не закладываем заранее; это отдельное будущее расширение.

Поле `colors` должно отражать фактические основные цвета темы:

- background;
- headings;
- text;
- link.

Важно: `theme-info.php` не должен становиться вторым источником истины. Он должен генерироваться из той же модели, что и CSS.

## 10. Превью и установка

Генератор должен давать пользователю уверенность, что скачанная тема поведёт себя так же, как превью.

Для этого нужно:

- использовать в превью ту же структуру переменных, что и в реальной теме;
- не рисовать отдельный “макет ради макета”, который не похож на Эгею;
- показывать состояния, на которых тема чаще всего ломается:
  - ссылки;
  - вторичный текст;
  - формы;
  - длинные заголовки;
  - мобильный вид;
  - светлый и тёмный фон, если позже появится dark mode.

В MVP можно использовать статический демо-контент, но он должен быть похож на реальный блог.

## 11. Экспорт

В первой версии проект должен работать без backend.

Желательный сценарий:

- все файлы генерируются в браузере;
- архив собирается на стороне клиента;
- скачивание начинается по нажатию кнопки;
- проект можно разместить на статическом хостинге.

Экспорт должен выдавать:

```text
theme-name.zip
└── theme-name/
    ├── theme-info.php
    └── styles/
        └── main.css
```

Позже можно добавить отдельный экспорт JSON-настроек темы, но в MVP это не обязательно.

## 12. Интерфейс

Интерфейс должен быть рабочим инструментом, а не лендингом.

Основной экран:

- слева панель настроек;
- справа большое превью;
- сверху или рядом название темы;
- внизу или в панели кнопка скачивания.

Контролы должны соответствовать типу данных:

- color picker для цветов;
- select для набора шрифтов;
- slider или stepper для размеров;
- segmented control для нескольких режимов, если он понадобится позже;
- toggle для бинарных параметров, если они появятся позже.

Не нужно превращать интерфейс в иллюстративный маркетинговый сайт. Он должен быть спокойным, плотным и удобным для многократных проб.

Интерфейс и превью должны сразу быть резиновыми и корректно работать на узких экранах. Отдельный переключатель мобильного превью в MVP не нужен: пользователь при необходимости может сузить окно браузера.

## 13. Шрифты

Изначальная идея упоминала Google Fonts, но для MVP это нужно ограничить.

В первой версии использовать только системные веб-безопасные шрифты, доступные на основных платформах, и показывать пользователю реальные названия шрифтов.

Причина простая: если шрифт не является системным и не поставляется вместе с темой, его наличие у пользователя нельзя гарантировать. Поэтому в MVP тема не должна зависеть от внешних файлов шрифтов или платных гарнитур.

В MVP тема не включает шрифтовые файлы и не подключает внешние шрифты.

Позже можно добавить:

- Google Fonts как отдельную фичу с генерацией подключения;
- отдельную подборку бесплатных кириллических open-source шрифтов;
- скачивание self-hosted font files вместе с темой для таких шрифтов.

## 14. Тёмная тема

Тёмную тему в MVP не делать.

Но архитектуру нужно подготовить так, чтобы позже можно было:

- редактировать светлую и тёмную палитры отдельно;
- включать `supports_dark_mode`;
- генерировать блок `@media (prefers-color-scheme: dark)`;
- при необходимости автоматически предлагать тёмный вариант из светлого.

## 15. Валидация качества темы

В MVP не делать автоматических предупреждений о качестве палитры и контрасте.

Позже можно добавить неблокирующие предупреждения про:

- читаемость текста на фоне;
- читаемость ссылок;
- заметность hover;
- различимость вторичного текста;
- достаточный контраст у полей ввода;
- отсутствие одинаковых цветов там, где состояния должны различаться.

## 16. Архитектурные требования

Проект должен быть устроен так, чтобы UI и генерация файлов не были смешаны.

Использовать:

- Vite;
- Vue;
- JavaScript без TypeScript.

По смыслу нужны отдельные части:

- модель темы;
- вычисление производных значений;
- генерация CSS;
- генерация `theme-info.php`;
- сборка архива;
- демо-контент для превью;
- UI-слой;
- валидаторы.

Важно, чтобы:

- состояние темы было сериализуемым;
- генераторы файлов можно было тестировать отдельно от UI;
- превью брало значения из той же модели, что и экспорт;
- добавление новых контролов не требовало переписывать весь проект.

Превью должно строиться на зафиксированной версии реальной разметки и контракта CSS-переменных Эгеи. В проекте нужно явно хранить, с какой версией Эгеи это превью сверено, и поддерживать короткий maintenance checklist на случай обновления движка:

- проверить список CSS-переменных `plain`;
- проверить значимые куски layout-разметки;
- проверить, не появились ли новые состояния, которые нужно показать в превью;
- при необходимости обновить эталонную разметку и список переменных.

## 17. Что не нужно делать в первой версии

Не делать в MVP:

- редактор произвольного CSS;
- редактор PHP-шаблонов;
- импорт существующих тем;
- поддержку не-`plain` баз;
- публикацию тем в общую галерею;
- аккаунты;
- backend;
- хранение пользовательских тем на сервере;
- полноценный каталог Google Fonts;
- кастомные изображения, логотипы и иконки;
- конструктор layout-шаблонов;
- visual diff между темами;
- автоматическую установку темы в живую Эгею;
- сложную систему пресетов с маркетинговыми названиями.

Сначала нужно доказать основной сценарий: человек без ручного кода может собрать и установить рабочую тему.

## 18. Возможные будущие расширения

После MVP можно добавить:

- тёмную палитру;
- сохранение состояния темы в URL;
- экспорт и импорт JSON-конфига;
- набор хороших пресетов;
- импорт существующей темы и продолжение редактирования;
- мультиязычность интерфейса, начиная с русского языка;
- Google Fonts;
- отдельную подборку бесплатных кириллических open-source шрифтов;
- self-hosted fonts для таких шрифтов;
- ручной advanced mode для всех CSS-переменных;
- загрузку собственного логотипа;
- галерею тем;
- шаринг ссылкой;
- режим сравнения с `plain`;
- предпросмотр нескольких типов страниц Эгеи;
- проверку темы на настоящем экземпляре Эгеи через специальный режим превью;
- выбор целевой версии Эгеи для генерации и превью;
- генерацию README для темы;
- экспорт темы как GitHub-ready папки.

## 19. Неопределённости, которые нужно решить перед реализацией

До начала разработки осталось отдельно выбрать:

1. На каком именно демо-контенте строить превью.
2. Какой минимальный maintenance process использовать для обновления превью при изменениях в Эгее.

## 20. Definition of Done для MVP

MVP считается готовым, если:

- генератор открывается в браузере;
- пользователь может менять основные цвета темы;
- пользователь может менять хотя бы базовые типографические параметры;
- изменения сразу видны в превью;
- превью достаточно полно показывает типичный блог Эгеи;
- пользователь может задать имя темы;
- по кнопке скачивается ZIP-архив с корректной папкой темы;
- в архиве есть валидные `theme-info.php` и `styles/main.css`;
- тема наследуется от `plain`;
- после загрузки в Эгею тема появляется в списке тем;
- после включения темы блог выглядит в целом так же, как в превью генератора;
- CSS и `theme-info.php` строятся из одного состояния, без ручного рассогласования;
- ошибки в базовых данных показываются пользователю до скачивания;
- архитектура позволяет без переписывания ядра добавить dark mode, пресеты, Google Fonts и импорт/экспорт конфигурации позже.

## 21. Критерий качества

Хороший результат — это не “много ручек”.

Хороший результат такой:

- человек без знания CSS быстро собирает тему, которая выглядит цельно;
- превью честно показывает будущий результат;
- скачанная папка сразу работает в Эгее;
- у темы есть характер, но интерфейс генератора не заставляет разбираться в устройстве движка;
- первая версия остаётся небольшой и расширяемой;
- следующий шаг после MVP очевиден: не чинить основу, а добавлять новые степени свободы.
