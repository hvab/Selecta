# Подготовительный трек перед продолжением Stage 1

Цель: один раз привести проект в рабочее бытовое состояние, чтобы дальше не смешивать
инфраструктурные правки с этапами MVP из старого `plan.md`.

## Решения по tooling

Для `Selecta` подходит пресет `base`, потому что проект на Vue + JavaScript, без React.

Сделано:

- [x] скопировать `eslint.config.cjs` из базового пресета;
- [x] скопировать `prettier.config.cjs`;
- [x] скопировать `.stylelintrc.cjs`;
- [x] скопировать `.editorconfig`;
- [x] скопировать `.browserslistrc`;
- [x] установить совместимые зависимости базового пресета:
  - `eslint`;
  - `@eslint/js`;
  - `globals`;
  - `prettier`;
  - `eslint-plugin-prettier`;
  - `eslint-config-prettier`;
  - `stylelint`;
  - `stylelint-config-standard`;
  - `stylelint-config-recess-order`;
  - `stylelint-order`.

Не делаем:

- пресет `react`;
- `eslint-plugin-react`;
- пакет `dotfiles25` как зависимость проекта;
- копирование `.gitignore` как есть: шаблон рассчитан на широкий набор Node-стеков, а в `Selecta`
  нужен короткий Vite-ориентированный вариант.

Также решено:

- [x] какие npm-скрипты добавить для lint/format-проверок;
- [x] оставлять ли stylelint уже сейчас как обязательную команду, пока в проекте только один простой CSS-файл.

Дополнительно для самого `Selecta`:

- [x] добавить `eslint-plugin-vue`, чтобы ESLint проверял `.vue`-файлы;
- [x] добавить `postcss-html`, чтобы Stylelint корректно читал стили внутри `.vue`-файлов.

## Подзадачи

### 1. Базовый запуск проекта

- [x] Обновить `README.md`:
  - что это за проект;
  - как установить зависимости;
  - как запускать разработку;
  - какой локальный URL открыть.
- [x] Сделать `npm start` основной командой запуска разработки.
- [x] При необходимости сохранить `npm run dev` как совместимый алиас, если это не создаёт лишнего шума.

Результат: новый человек может поднять проект, не заглядывая в исходники.

### 2. Базовые инструменты качества

- [x] Скопировать применимые конфиги из базового пресета `dotfiles25`.
- [x] Добавить только применимые зависимости.
- [x] Добавить npm-скрипты для ESLint, Prettier и Stylelint.
- [x] Проверить, что конфиги реально запускаются на текущих файлах проекта.

Результат: проект получает единый стиль и проверяемые команды без React-специфики.

### 3. Упростить `.gitignore`

- [x] Оставить только реально нужное для текущего Vite-проекта:
  - `node_modules/`;
  - `dist/`;
  - `.vite/`;
  - логи;
  - локальные env-файлы;
  - кэши линтеров;
  - `AGENTS.local.md`.
- [x] Убрать записи под неиспользуемые стеки и инструменты.

Результат: `.gitignore` соответствует самому проекту, а не универсальному шаблону.

### 4. UI-архитектура генератора

- [x] Зафиксировать локальное решение в `UI-ARCHITECTURE.md`:
  - превью живёт по реальному контракту Эгеи и теме `plain`;
  - оболочка генератора остаётся отдельным стабильным UI;
  - в MVP используются нативные контролы и локальные стили;
  - headless-примитивы допустимы позже точечно для сложных взаимодействий.

Результат: у проекта есть собственное решение по UI, не зависящее от чужой дизайн-системы.

### 5. Автоматические тесты

- [x] Не добавлять тестовую инфраструктуру только ради scaffold-этапа.
- [x] Начать автоматические тесты с первой чистой логики генерации:
  - минимальный `styles/main.css`;
  - минимальный `theme-info.php`;
  - затем производные значения, validation и ZIP.

Результат: тесты появляются там, где уже есть стабильная чистая логика, а не раньше ради формальности.

### 6. Вернуться к основному плану

- [x] После завершения подготовительного трека обновить старый `PROGRESS.md`.
- [x] Вернуться к `Stage 1.2: Initial serializable theme state model`.

## Предлагаемый порядок

1. `README.md` + `npm start`.
2. Копирование нужных конфигов из `dotfiles25`.
3. Упрощение `.gitignore`.
4. Зафиксировать локальную UI-архитектуру.
5. Зафиксировать момент старта автоматических тестов.
6. Короткая сверка состояния и возврат к `Stage 1.2`.

## Границы

- Этот трек не должен менять смысл MVP из `SPEC.md`.
- Здесь не начинаем реальное превью Эгеи, controls, ZIP или валидацию.
- Не переносим чужую дизайн-систему как готовое решение; внешние системы можно использовать только как источник отдельных уместных идей.
