# План итерационной реализации MVP

## Общий подход

Работаем маленькими итерациями по `SPEC.md`.

Главная формула каждой итерации:

```text
Сначала проверь текущее состояние по SPEC.md, потом реализуй только следующий маленький этап.
```

После каждой завершённой итерации:

1. Проверить соответствие `SPEC.md`.
2. Коротко описать, что сделано.
3. Указать, как проверить результат локально.
4. Указать известные ограничения.
5. Предложить следующий маленький шаг.

## Зафиксированные решения

- Стек: Vite, Vue, JavaScript.
- Без TypeScript.
- Без backend.
- Без тёмного режима в MVP.
- Без пресетов в MVP.
- Без JSON-экспорта в MVP.
- Без автоматических предупреждений о контрасте в MVP.
- Только системные веб-безопасные шрифты в MVP.
- Весь пользовательский интерфейс MVP только на английском языке.
- Без i18n-инфраструктуры в MVP.
- Google Fonts позже отдельной фичей.
- Отдельная подборка бесплатных кириллических open-source шрифтов позже отдельной фичей.
- Генерируем только дочерние темы на основе `plain`.
- Экспортируем ZIP с папкой темы внутри.
- Превью строим максимально близко к реальной Эгее, на основе настоящей разметки и тех же CSS-переменных.
- Превью и интерфейс генератора визуально разделены: превью живёт по контракту Эгеи, а оболочка генератора остаётся отдельным стабильным UI.
- Интерфейс и превью сразу должны быть responsive.

## Подготовка

`SPEC.md` — главный источник требований.

Перед началом реализации нужно отдельно зафиксировать:

1. Какой именно демо-контент используется в превью.
2. С какой версией Эгеи сверено превью.
3. Как выглядит короткий checklist для обновления превью при изменениях в Эгее.

## Стартовый промт

```text
Ты работаешь над новым frontend-проектом Selecta: генератором тем для Эгеи.

В корне проекта есть файл SPEC.md. Сначала внимательно прочитай его целиком и используй как главный источник требований.

Задача: реализовать MVP по этой спецификации.

Важные ограничения:
- Используй Vite, Vue и обычный JavaScript.
- Не используй TypeScript.
- Не добавляй backend.
- Не добавляй dark mode в MVP.
- Не добавляй пресеты в MVP.
- Не добавляй JSON export/import в MVP.
- Не добавляй автоматические предупреждения о контрасте в MVP.
- Для шрифтов в MVP используй только системные веб-безопасные варианты.
- Весь пользовательский интерфейс MVP делай только на английском языке.
- Не добавляй i18n-инфраструктуру в MVP.
- Генерируй только дочерние темы на основе plain.
- Превью должно быть максимально близким к реальной Эгее и использовать тот же контракт CSS-переменных.
- Не уходи в сложную архитектуру раньше времени.

Работай итеративно.

Сначала:
1. Изучи SPEC.md.
2. Коротко перескажи, что нужно сделать.
3. Предложи минимальный план реализации на 5–8 шагов.
4. Укажи, какие файлы и модули собираешься создать.
5. После этого начинай реализацию только первого логического этапа.

Первый этап должен дать базовый каркас проекта и контракт генерации темы:
- Vite + Vue + JavaScript;
- базовую модель theme state;
- генерацию минимального styles/main.css;
- генерацию минимального theme-info.php;
- минимальный экран, где видны текущие значения темы и результат генерации как текст.

После завершения первого этапа:
- перечисли, что сделано;
- укажи, как проверить результат локально;
- перечисли известные ограничения;
- предложи следующий маленький шаг.
```

## Этап 1. Каркас и контракт темы

Цель: быстро доказать, что из одного состояния темы можно надёжно получить оба выходных файла.

Сделать:

- создать Vite + Vue проект;
- описать начальную модель темы;
- вынести генерацию CSS отдельно от UI;
- вынести генерацию `theme-info.php` отдельно от UI;
- показать на экране текущее состояние темы и содержимое генерируемых файлов.

Тесты:

- не поднимать тестовую инфраструктуру ради самого каркаса;
- начать автоматические тесты, когда появится первая чистая логика генерации:
  - минимальный `styles/main.css`;
  - минимальный `theme-info.php`;
  - позже производные значения и валидация.

Не делать пока:

- полноценное превью;
- панель контролов;
- ZIP;
- сложный UI;
- шрифтовые настройки сверх минимального дефолта.

## Этап 2. Эталонное превью Эгеи

Цель: собрать честную основу preview surface до того, как появятся крутилки.

Сделать:

- выбрать статический демо-контент;
- собрать превью на основе реальных кусков разметки Эгеи;
- подключить те же CSS-переменные, которые будут попадать в экспортируемую тему;
- зафиксировать, с какой версией Эгеи сверено превью;
- описать maintenance checklist для будущих обновлений.

Не делать пока:

- экспорт;
- пресеты;
- dark mode;
- отдельный mobile toggle.

## Этап 3. Панель основных контролов

Цель: получить главный рабочий цикл генератора.

Сделать:

- контролы основных цветов;
- выбор системного шрифта интерфейса;
- выбор системного шрифта текста заметок;
- размер основного текста;
- масштаб заголовков;
- межстрочный интервал;
- max width;
- боковые отступы;
- live update превью;
- live update генерируемого CSS и `theme-info.php`.

Важно:

- все изменения идут через одну модель состояния;
- превью и экспортные файлы не расходятся;
- интерфейс остаётся responsive.

## Этап 4. Метаданные темы

Цель: довести пользовательские поля темы до состояния, пригодного для экспорта.

Сделать:

- поле отображаемого имени темы;
- автогенерацию технического имени папки;
- возможность вручную исправить техническое имя;
- валидацию обязательных полей;
- защиту от недопустимых символов в имени папки;
- отображение ошибок до скачивания.

## Этап 5. Скачивание темы

Цель: завершить основной пользовательский сценарий.

Сделать:

- сборку ZIP на клиенте;
- структуру архива:

```text
theme-name.zip
└── theme-name/
    ├── theme-info.php
    └── styles/
        └── main.css
```

- скачивание по одной кнопке;
- понятное имя архива.

## Этап 6. Проверка в живой Эгее

Цель: убедиться, что генератор выдаёт не просто красивые файлы, а реально рабочую тему.

Сделать:

- установить скачанную тему в локальную Эгею;
- проверить, что она появляется в списке тем;
- проверить, что после включения блог выглядит в целом так же, как в превью;
- сверить `theme-info.php` и фактические CSS-переменные;
- зафиксировать найденные расхождения и исправить их.

Это обязательный этап MVP, а не необязательная полировка.

## Этап 7. Финальная полировка MVP

Цель: довести первую версию до цельного инструмента.

Сделать:

- убрать шероховатости в UX;
- проверить responsive layout;
- проверить поведение на длинных названиях темы;
- проверить пустые и пограничные значения;
- привести README к фактическому сценарию запуска и использования;
- обновить `PROGRESS.md`, если он к этому моменту уже заведён.

## Рекомендуемый порядок модулей

```text
src/
  app/
  preview/
  theme/
    model.js
    css.js
    themeInfo.js
    zip.js
    validation.js
  components/
```

Точные имена можно поменять по ходу реализации, но разделение по ответственности нужно сохранить:

- состояние темы;
- производные значения;
- генерация файлов;
- превью;
- UI.

## Чего не делать до завершения MVP

- Не добавлять dark mode.
- Не добавлять пресеты.
- Не добавлять Google Fonts.
- Не добавлять bundled open-source fonts.
- Не добавлять JSON export/import.
- Не добавлять автоматические contrast warnings.
- Не добавлять импорт существующих тем.
- Не добавлять произвольный CSS editor.
- Не добавлять несколько типов layout.
- Не добавлять i18n-инфраструктуру.

## После MVP

Логичный порядок следующих фич:

1. Пресеты.
2. Google Fonts.
3. Сохранение состояния темы в URL или JSON.
4. Тёмный режим.
5. Подборка бесплатных кириллических open-source шрифтов с включением файлов в тему.
6. Импорт существующих тем.
7. Мультиязычность интерфейса, начиная с русского языка.
